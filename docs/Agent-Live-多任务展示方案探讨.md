# Agent Live 多任务展示方案探讨

> 当多个任务同时执行时，LIVE 视图如何展示？

---

## 一、现状

| 组件 | 数据来源 | 多任务支持 |
|------|----------|------------|
| **EXEC** (ExecutionTimelineView) | `vm.tasks` + `vm.selectedTaskId` | ✅ 左侧任务列表，点击切换 |
| **LIVE** (AgentLiveView) | `vm.currentTaskData ?? vm.xxx` | ⚠️ 仅显示「当前选中」任务的 Live |

- `currentTaskData` = 选中任务 或 运行中任务 或 最近更新任务
- LIVE 无任务列表，用户需切到 EXEC 选任务再回 LIVE，体验割裂

---

## 二、方案对比

### 方案 A：复用 EXEC 布局（左列表 + 右 Live）

```
┌─────────────────────────────────────────────────────────────┐
│  LIVE                                                       │
├──────────────┬──────────────────────────────────────────────┤
│ [任务列表]    │  [选中任务的 Live 终端]                       │
│ ─────────    │  ─────────────────────                       │
│ ● 任务A 运行 │  🧠→[文件]  READ FILE /path/...               │
│ ○ 任务B 完成 │  [STREAM] FILE_OPERATIONS ...                 │
│ ○ 任务C 完成 │  [LOG] ✓ READ FILE → /path/...               │
│ ─────────    │                                              │
│ [单任务][全部]│                                              │
└──────────────┴──────────────────────────────────────────────┘
```

**优点**：与 EXEC 一致，用户已熟悉；实现简单（复用 TaskListSidebar）  
**缺点**：一次只能看一个任务的 Live；与 EXEC 功能重叠

---

### 方案 B：多任务并行展示（分栏 / 分块）

```
┌─────────────────────────────────────────────────────────────┐
│  LIVE · 3 任务并行                                           │
├─────────────────────┬─────────────────────┬─────────────────┤
│ 任务A (运行中)       │ 任务B (运行中)       │ 任务C (已完成)   │
│ ─────────────────   │ ─────────────────   │ ─────────────   │
│ 🧠→[文件]           │ 🧠→[网络搜索]        │ ✓ 完成          │
│ READ FILE ...       │ WEB_SEARCH ...       │                 │
│ [STREAM] ...        │ [STREAM] ...        │                 │
└─────────────────────┴─────────────────────┴─────────────────┘
```

**优点**：一眼看到所有任务状态；适合「控制室」感  
**缺点**：空间有限，每栏窄；2–3 个任务尚可，更多会挤

---

### 方案 C：活动脉动 + 主从

```
┌─────────────────────────────────────────────────────────────┐
│  [●A] [●B] [○C]  ← 顶部：任务状态点，点击切换主视图           │
├─────────────────────────────────────────────────────────────┤
│  主视图：选中任务的完整 Live 终端                             │
│  🧠→[文件]  READ FILE ...                                   │
│  [STREAM] [LOG] ...                                          │
├─────────────────────────────────────────────────────────────┤
│  其他任务：单行摘要（可展开）                                  │
│  B: 🧠→[网络搜索] 正在执行...                                 │
│  C: ✓ 已完成                                                 │
└─────────────────────────────────────────────────────────────┘
```

**优点**：主次分明；其他任务有存在感但不抢主视图  
**缺点**：需设计摘要行样式

---

### 方案 D：轮播 / 自动切换

- 每 N 秒自动切换到下一个运行中任务
- 或：最近有事件的任务自动置顶

**优点**：无需手动切换  
**缺点**：可能错过关键瞬间；用户失去控制感

---

### 方案 E：统一流 + 任务标签

```
┌─────────────────────────────────────────────────────────────┐
│  [STREAM] 全部任务合并                                        │
├─────────────────────────────────────────────────────────────┤
│  [A] 🧠→[文件]  READ FILE ...                                │
│  [B] 🧠→[网络搜索]  WEB_SEARCH ...                           │
│  [A] [STREAM] FILE_OPERATIONS ...                             │
│  [B] [STREAM] WEB_SEARCH ...                                 │
└─────────────────────────────────────────────────────────────┘
```

**优点**：时间顺序清晰；类似「全部」时间轴  
**缺点**：多任务交织，单任务上下文不连贯；与 EXEC 的「全部」模式重复

---

## 三、推荐组合

| 阶段 | 方案 | 说明 |
|------|------|------|
| **MVP** | **方案 A** | LIVE 加左侧任务列表，与 EXEC 布局统一；选中任务显示其 Live |
| **增强** | **方案 C** | 顶部任务状态点 + 底部其他任务摘要行，增强多任务感知 |
| **可选** | **方案 B** | 2–3 任务时提供「分栏」视图切换 |

---

## 四、MVP 实现要点（方案 A）

1. **布局**：`HSplitView { TaskListSidebar | AgentLiveView }`，与 EXEC 一致  
2. **数据**：AgentLiveView 继续用 `currentTaskData`，选中任务由 TaskListSidebar 驱动  
3. **复用**：TaskListSidebar 可从 ExecutionTimelineView 抽成共享组件，或 LIVE 内嵌简化版  
4. **executionLogs**：需按任务分桶；当前 `MonitoringViewModel.executionLogs` 若为全局合并，需在 `TaskMonitorData` 中增加 `executionLogs`

---

## 五、数据现状与缺口

| 数据 | 当前存储 | 多任务支持 |
|------|----------|------------|
| actionLogs | `TaskMonitorData.actionLogs` | ✅ 按 task_id 分桶 |
| executionLogs | `MonitoringViewModel.executionLogs` 全局 | ❌ 未分桶，来自 `agent.$executionLogs` |
| taskProgress, llmStreamingText 等 | `TaskMonitorData` | ✅ 按 task 分桶 |

**结论**：`execution_log` 事件若未带 `task_id`，多任务时 [STREAM] 会混在一起。需后端在 `execution_log` 中增加 `task_id`，且 Mac 端按 task 分桶写入 `TaskMonitorData.executionLogs`（或新建字段）。

---

## 六、待确认问题

1. **executionLogs 分桶**：后端 `execution_log` 是否含 `task_id`？Mac 端是否需改造为按任务存储？
2. **LIVE 与 EXEC 的定位**  
   - LIVE：偏「氛围 / 实时感」  
   - EXEC：偏「复盘 / 时间轴」  
   - 是否允许两者布局高度一致（左列表 + 右内容）？

3. **任务列表数据源**（若采用方案 A）  
   - 用 `vm.tasks`（来自 monitor_event）  
   - 或 `vm.activeTaskList`（来自 /monitor/active-tasks 轮询）  
   - 两者可能不同步，需明确主数据源
