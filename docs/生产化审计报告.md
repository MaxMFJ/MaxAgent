## 2. CRITICAL — 必须立即修复

### C1. 内存泄漏 — 调度器任务字典无限增长 [已修复 ✅]

**文件：** `backend/services/duck_task_scheduler.py`

**问题：** 4 个核心字典(`_tasks`, `_callbacks`, `_task_sessions`, `_timeout_handles`)只在任务成功完成时清理。以下场景导致条目永不释放：
- 任务创建后从未被分配（无可用 Duck）
- 回调执行失败  
- 服务重启后从磁盘恢复的旧任务

**影响：** 持续运行 1-2 周后，内存占用可达数百 MB 并持续增长。长期运行会导致 OOM。

**修复方案：** ✅ 已实施
- 在 `__init__` 中添加 `_task_ttl_seconds = 3600`（1小时 TTL）和 `_last_cleanup_time` 计时器
- 新增 `_cleanup_expired_tasks()` 方法：每 5 分钟扫描一次，清理已终结（COMPLETED/FAILED/CANCELLED）且超过 TTL 的任务
- 清理范围：`_tasks`、`_callbacks`、`_task_sessions`、`_timeout_handles` 四个字典 + 持久化 JSON 文件
- 集成到 watchdog 循环中自动运行

---

### C2. 内存泄漏 — DAG 执行记录永不清理 [已修复 ✅]

**文件：** `backend/services/duck_task_dag.py`

**问题：** `_executions` 和 `_task_to_dag` 字典永不清理已完成的 DAG。

**影响：** 每次创建 DAG 都增加内存占用，长期运行后积累大量无用数据。

**修复方案：** ✅ 已实施
- 在 `__init__` 中添加 `_execution_ttl = 1800`（30分钟 TTL）
- 新增 `_schedule_dag_cleanup()` 方法：DAG 完成/取消后延迟 TTL 时间自动清理
- 清理范围：`_executions` 中的 DAG 记录 + `_task_to_dag` 中关联的任务映射
- 在 `_check_dag_completion` 和 `cancel` 末尾自动触发

---

### C3. DAG 串行任务并发执行 [已修复 ✅]

**文件：** `backend/services/duck_task_dag.py`

**问题：** 串行任务导致所有子 Duck 同时执行（本次审计前已修复），根因包括：
1. `_schedule_ready_nodes` 提交后未设 `execution_emitted` 标志
2. `normalize_dag_dependencies` 有部分 depends_on 时完全跳过自动串行
3. `_on_node_complete` 缺乏并发保护

**状态：** ✅ 已修复

---

### C4. ConnectionManager pending_buffers 泄漏 [已修复 ✅]

**文件：** `backend/connection_manager.py`

**问题：** `_pending_buffers` 字典中的 session 条目在连接断开后不清理。大量唯一 session 会导致无限增长。

**修复方案：** ✅ 已实施
- 新增 `_sweep_expired_buffers()` 方法：每 60 秒扫描一次，清理所有消息已过期（超过 `_BUFFER_EXPIRE_SECS`）的 session 缓冲
- 在 `_buffer_message()` 中自动触发周期性清理
- 同时清理空 buffer 和全部过期的 buffer

## 3. HIGH — 生产化前必须解决

### H1. WebSocket 重连时不刷新缓冲消息 [无需修复 — 已有实现]

**文件：** `backend/ws_handler.py`

**复查结果：** `/ws` handler 在连接建立后已调用 `flush_pending(session_id, websocket)`（第 1968 行），session 切换时也会刷新（第 2004 行）。此问题不存在。

---

### H2. LLM 调用无指数退避重试 [已修复 ✅]

**文件：** `backend/agent/llm_client.py`

**问题：** API 返回 5xx 或网络错误时直接返回错误，没有重试逻辑。

**修复方案：** ✅ 已实施
- 新增 `_call_with_retry()` 方法，支持指数退避 + 随机抖动（1s → 2s → 4s → 8s）
- 最多重试 3 次，非 5xx 错误（401/403/400/quota/context_length）立即失败不重试
- 同时应用于 `chat()` 和 `chat_stream()` 两个 API 调用路径
- 兼容现有 `timeout_policy` 超时控制

---

### H3. 文件工具缺少路径限制 [已修复 ✅]

**文件：** `backend/tools/file_tool.py`

**问题：** `_normalize_path()` 只做路径展开，不检查最终路径是否在允许范围内。

**修复方案：** ✅ 已实施
- 新增 `_check_path_allowed()` 函数，使用 `os.path.realpath` 解析 symlink 后检查
- 允许范围：`~/`、`/tmp`、`/var/folders`（macOS 临时目录）
- 在 `execute()` 入口 + move/copy 的 destination 路径均做校验
- 路径越界抛出 `PermissionError`

---

### H4. 终端工具危险命令拦截不完整 [已修复 ✅]

**文件：** `backend/tools/terminal_tool.py`

**问题：** 使用黑名单方式拦截危险命令，仅做简单子串匹配，容易被绕过。

**修复方案：** ✅ 已实施
- 危险模式从纯字符串改为正则表达式，使用 `re.compile()` 预编译
- 新增模式：`curl|sh`、`wget|sh`、`sudo rm`、`shutdown`、`reboot`、`diskutil erase` 等
- 使用 `\b` 词边界、`\s+` 空格匹配，防止空格/符号变体绕过
- 所有测试用例通过验证

---

### H5. API Key 明文存储 [已修复 ✅]

**文件：** `backend/config/llm_config.py` + 新增 `backend/config/keychain_helper.py`

**问题：** LLM API Key 以明文 JSON 存储在 `data/llm_config.json`，无加密。

**修复方案：** ✅ 已实施
- 新增 `keychain_helper.py`：封装 macOS `security` CLI 工具，提供 `keychain_store/load/delete`
- `save_llm_config()` 保存 API Key 时优先存入 Keychain，JSON 中仅存 `***keychain***` 掩码
- `get_persisted_api_key()` 加载时优先从 Keychain 读取，跳过掩码值
- 向后兼容：Keychain 不可用时退回明文 JSON

### H6. Duck 崩溃后任务恢复延迟过长 [已修复 ✅]

**文件：** `backend/services/duck_task_scheduler.py`

**问题：** Lease 超时阈值为 120s。Duck 崩溃后最长需要 130s 才能重新调度任务。

**修复方案：** ✅ 已实施
- `lease_timeout` 从 120s 降至 60s（崩溃检测最长 70s）
- `lease_scan_interval` 已是 10s（无需调整）

---

### H7. 配置更新无文件锁 [已修复 ✅]

**文件：** `backend/config/llm_config.py`

**问题：** 多进程/多线程并发写入 JSON 配置文件可能导致数据损坏。

**修复方案：** ✅ 已实施
- 使用原子写入：`tempfile.mkstemp()` 写临时文件 → `os.replace()` 原子替换
- 写入失败时自动清理临时文件
- `os.replace()` 在 POSIX 系统上是原子操作

---

### H8. 任务持久化 orphan_timer 竞态条件 [已修复 ✅]

**文件：** `backend/task_persistence.py`

**问题：** `_orphan_timers` 字典在重复断开连接时存在竞态：旧 timer 的 `finally` 块会错误清理新 timer 的引用。

**修复方案：** ✅ 已实施
- 启动新 timer 前先 `pop` 旧 timer（而非仅 `cancel`）
- `finally` 块中检查 `asyncio.current_task()` 是否仍是注册的 task，只有匹配时才清理

## 4. MEDIUM — 生产化后尽快解决

### M1. 无请求/Token 用量限额 [已修复 ✅]

**问题：** 单个用户/session 可无限制调用 LLM，没有每日/每月 token 上限或请求频率限制。生产环境下可能产生巨额 API 费用。

**修复方案：** ✅ 已实施
- `UsageTracker` 新增 `check_rate_limit()` 方法，基于 1 分钟滑动窗口检查 RPM/TPM
- 配置：环境变量 `MACAGENT_RPM_LIMIT`（默认 60）、`MACAGENT_TPM_LIMIT`（默认 200000）
- 集成到 `LLMClient._call_with_retry()` 中，每次 API 调用前预检

### M2. 无审计日志持久化 [已存在 ✅]

**问题：** 运行时日志仅写入 stdout/文件，无结构化审计日志。

**现状：** ✅ 已有完整实现
- `audit_service.py` 提供 `append_audit_event()` 结构化审计日志
- 17 处调用点覆盖工具调用、文件操作、命令执行等关键路径
- 支持 `AUDIT_LOG_MAX_SIZE_MB` 磁盘配额自动轮转

### M3. 群聊服务无持久化 [已存在 ✅]

**问题：** 群聊数据只存内存中，服务重启后群聊记录丢失。

**现状：** ✅ 已有完整实现
- `group_chat_service.py` 实现 JSON 文件持久化（`data/group_chats/`）
- 支持消息、会话、元数据的完整持久化和恢复

### M4. 无用户身份认证体系 [架构级 — 跳过]

**问题：** WebSocket 连接使用 token 认证但无用户概念，任何持有 token 的人可执行任何操作。

**状态：** 架构级变更，需整体规划后实施

### M5. 无任务优先级队列 [已修复 ✅]

**问题：** DAG 节点虽有 priority 字段，但 ready queue 是标准 FIFO Queue，不按优先级排序。

**修复方案：** ✅ 已实施
- 新增 `PriorityReadyQueue` 类，包装 `asyncio.PriorityQueue`
- 使用 `(priority, seq, item)` 三元组排序，priority 越小越优先，seq 保证 FIFO 平局
- 对外 API 与 `asyncio.Queue` 兼容（put/get/qsize/empty/full）
- `get_ready_queue()` 改为创建 `PriorityReadyQueue` 实例
- 快照持久化/恢复兼容新格式

### M6. 自愈诊断引擎缺少单元测试 [测试级 — 跳过]

**问题：** 自愈功能的修复策略未经充分测试，错误的修复可能导致连锁故障。

**状态：** 测试级任务，不影响运行时稳定性

### M7. Stream 流关闭不可靠 [已存在 ✅]

**文件：** `backend/agent/llm_client.py`

**问题：** LLM 流式响应在异常路径上 `aclose()` 可能静默失败，导致连接池耗尽。

**现状：** ✅ 已有完整实现
- `chat_stream()` 使用 `try/finally` 块确保 `stream.aclose()` 在所有路径（正常/异常/超时）都被调用
- 支持 `aclose()` 和 `close()` 两种接口

### M8. 无数据目录大小管理 [已修复 ✅]

**文件：** `backend/data/` 目录

**问题：** 任务检查点、日志文件持续增长，无定期清理或轮转机制。

**修复方案：** ✅ 已实施
- 新增 `services/data_cleanup.py` 模块，后台每小时执行一次
- 管理 4 个目录的 TTL 清理策略：
  - `task_store/checkpoints/`: 7 天
  - `traces/`: 7 天
  - `usage_stats/`: 30 天
  - `episodes/`: 14 天
- 已与 main.py lifespan 集成（启动时开始，关停时停止）
- 已有管理的目录不受影响：`duck_tasks/`（1h TTL）、`audit/`（100MB 配额）

### M9. DAG 回调失败恢复过于激进 [已修复 ✅]

**文件：** `backend/services/duck_task_scheduler.py`

**问题：** 回调执行异常时直接调用 `_schedule_ready_nodes`，可能在错误状态下触发新任务。

**修复方案：** ✅ 已实施
- 回调失败时改用 `_on_node_complete()`（幂等入口）替代直接调用 `_schedule_ready_nodes`
- `_on_node_complete` 内部有完整的状态检查、锁保护和幂等性保证
- 防止在不一致状态下触发新的下游任务

- **静态分析：** 逐文件审查 backend/services/、backend/agent/、backend/tools/ 目录
- **搜索覆盖：** 关键词搜索：cron, schedule, timer, periodic, 定时, 周期, leak, race, lock, security, injection, overflow
- **数据流追踪：** 从 WebSocket 入口 → Agent 调度 → Duck 分发 → 结果回调全链路追踪
- **模式分析：** 识别 Dict 增长模式、try/except 静默模式、并发访问模式
