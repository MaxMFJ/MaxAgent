# v3.1 功能测试指南

本文说明如何运行 v3.1 单元测试以及如何手动验证各项功能。v3.1 实现内容见 [V3.1_PLAN.md](./V3.1_PLAN.md)。

## 一、单元测试

### 1.1 运行方式

在 **backend** 目录下执行其一即可：

```bash
cd backend

# 方式一：仅用标准库（推荐，无需安装 pytest）
python tests/test_v31.py

# 带详细输出
python tests/test_v31.py -v

# 方式二：使用 pytest（需先安装 pytest）
python -m pytest tests/test_v31.py -v
```

### 1.2 无完整依赖时

测试文件已对 `openai`、`numpy`、`httpx`、`psutil` 等做 mock，在**未安装** `requirements.txt` 全量依赖时也可跑通。若已创建虚拟环境并安装依赖，将使用真实模块。

### 1.3 覆盖范围

| 测试类 | 内容 |
|--------|------|
| **TestSafety** | `validate_command` 危险/安全命令；`validate_action_safe` 对 RUN_SHELL、WRITE_FILE、DELETE_FILE、CALL_TOOL(terminal)、CREATE_AND_RUN_SCRIPT 的拦截与放行 |
| **TestStructuredMemory** | `get_structured_history` 空/有日志；`summarize_history_for_llm` 空/有日志/截断 |
| **TestEscalation** | `_detect_repeated_failure` 无日志返回 NORMAL；多条同错误失败时 level ≥ 1 |
| **TestAppStateFlags** | v3.1 相关 FeatureFlag 存在且类型正确 |

## 二、手动功能验证

以下在**已启动后端 + 前端/客户端**的前提下，通过发真实任务观察行为。

### 2.1 结构化 Memory + 上下文压缩

- **开关**：`USE_SUMMARIZED_CONTEXT`（默认 true）
- **验证**：发起一次步数较多的自主任务（例如 15+ 步），在日志或调试输出中确认：
  - 传给 LLM 的上下文中使用的是「摘要 + 最近若干条」形式，而不是整段原始 action_log 拼接。
- **代码位置**：`autonomous_agent._generate_action` 中调用 `context.summarize_history_for_llm()`。

### 2.2 Goal 重述

- **开关**：`GOAL_RESTATE_EVERY_N`（默认 6）
- **验证**：长任务中每隔 N 步，prompt 中应出现 **【当前目标】原始任务: xxx**。
- **可选**：若开启 Plan-and-Execute，可同时观察 **【当前建议子目标】** 是否随步骤推进。

### 2.3 Plan-and-Execute

- **开关**：`ENABLE_PLAN_AND_EXECUTE`（默认 false，需显式开启）
- **环境变量**：`MACAGENT_ENABLE_PLAN_AND_EXECUTE=true`
- **验证**：
  1. 开启后发起一个多步骤任务，应先在流式输出中看到 `plan_created` 及子步骤列表。
  2. 执行过程中应看到「当前建议子目标」类提示。
  3. 当触发 escalation（见下）达到 FORCE_SWITCH 时，应出现 `plan_replanned` 及新计划。

### 2.4 Escalation 智能触发

- **开关**：`ESCALATION_FORCE_AFTER_N`、`ESCALATION_SKILL_AFTER_N`、`ESCALATION_SIMILARITY_THRESHOLD`
- **默认**：连续 2 次相似失败 → FORCE_SWITCH；连续 3 次 → SKILL_FALLBACK；相似度阈值 0.85
- **验证**：构造「连续多次相同或高度相似错误」的任务（例如故意让某工具/命令反复失败），观察：
  - 是否在连续 N 次后出现升级（切换模型或技能 fallback），
  - 以及是否触发 replan（若开启了 Plan-and-Execute）。

### 2.5 中途反思

- **开关**：`ENABLE_MID_LOOP_REFLECTION`、`MID_LOOP_REFLECTION_EVERY_N`（默认每 5 步）
- **验证**：长任务或连续失败时，在达到每 N 步或失败触发时，下一轮 prompt 中应出现 **【中途反思建议】**，且内容与当前困境相关。

### 2.6 安全校验

- **代码位置**：`agent/safety.py`（`validate_command`、`validate_path_for_write/delete`、`validate_action_safe`）
- **验证**：
  1. **RUN_SHELL**：下发 `rm -rf /`、`sudo rm`、格式化磁盘等，应被拦截并返回错误提示。
  2. **WRITE_FILE / DELETE_FILE**：路径为 `/etc/passwd`、系统关键目录等，应被拦截。
  3. **CALL_TOOL(terminal)**：同上，对终端类工具内部命令做相同校验。
  4. **CREATE_AND_RUN_SCRIPT**：脚本内容包含危险命令时，应被拦截。
- 安全通过时，正常写用户目录、执行 `ls`、`echo` 等应放行。

## 三、关键环境变量（v3.1）

| 变量名 | 含义 | 默认 |
|--------|------|------|
| `MACAGENT_USE_SUMMARIZED_CONTEXT` | 使用摘要+最近条目的上下文 | true |
| `MACAGENT_GOAL_RESTATE_EVERY_N` | 每 N 步重述目标 | 6 |
| `MACAGENT_ENABLE_PLAN_AND_EXECUTE` | 启用先规划再执行 | false |
| `MACAGENT_ENABLE_MID_LOOP_REFLECTION` | 启用中途反思 | true |
| `MACAGENT_MID_LOOP_REFLECTION_EVERY_N` | 每 N 步触发中途反思 | 5 |
| `MACAGENT_ESCALATION_FORCE_AFTER_N` | 连续 N 次相似失败后强制切换 | 2 |
| `MACAGENT_ESCALATION_SKILL_AFTER_N` | 连续 N 次后技能 fallback | 3 |
| `MACAGENT_ESCALATION_SIMILARITY_THRESHOLD` | 失败相似度阈值 (0~1) | 0.85 |

以上开关均在 `app_state.py` 中定义，并在 `agent/autonomous_agent.py` 的 `_generate_action` 及主循环（Goal 重述、中途反思、escalation）中接通。

## 四、常见现象说明

### 4.1 第一步就出现「finish + 纯文本总结」

**现象**：任务刚开始，步骤 1 就显示「finish」，总结是一大段模型说明（例如「我无法访问实时数据…」「建议你…」），没有执行任何 run_shell / call_tool 等动作。

**原因**（与 2.1 结构化 memory 无关）：

- 第一步时还没有历史，`summarize_history_for_llm` 只返回「任务 + 当前迭代」，不会改变首轮行为。
- 真实原因是：**远程模型返回了纯自然段文字**（没有输出要求的 JSON），解析失败后旧逻辑会把整段文字当作「纯文本 finish」兜底，于是出现「步骤 1: finish」。

**已做改进**（代码层）：

- 若当前是**第一步**（无历史）且「纯文本」较长（>200 字），**不再**将其视为 finish，而是当作解析失败，触发重试。
- 重试时会在首轮提示后追加：**「你必须只输出一个 JSON 动作（如 run_shell、call_tool），不要输出大段说明；即使认为难以完成，也请先输出一个尝试性动作的 JSON。」**

若仍出现第一步就 finish，可检查：系统提示中「禁止在未执行具体操作前就输出 finish」是否被模型遵守；必要时在任务描述中明确「请先执行至少一步操作（如搜索或执行命令）再总结」。

### 4.2 用 file_operations create 写报告后文件为空

**现象**：任务要求「生成 Markdown 报告并保存到桌面」，日志显示 `file_operations` 调用 `action: "create"` 且带很长 `content`，工具返回 success，但桌面上的文件是空白的。

**原因**：`file_operations` 的 `create` 此前只区分「创建目录」或「创建空文件」（`Path(path).touch()`），**没有处理 `content` 参数**，因此传入的正文被忽略，只生成了空文件。

**修复**：在 `backend/tools/file_tool.py` 中，当 `action == "create"` 且传入了 `content` 时，改为与 `write` 一致：先创建父目录，再写入文件内容；未传 `content` 时仍创建空文件。

## 五、测试通过标准

- **单元测试**：在 backend 下执行 `python tests/test_v31.py -v`，16 个用例全部通过。
- **手动测试**：按第二节逐项验证，关键路径（上下文压缩、Goal 重述、安全拦截、escalation）与预期一致即可认为 v3.1 功能测试通过。
- **第一步纯文本**：若模型第一步就回大段说明而不出 JSON，应触发重试并看到「你必须只输出一个 JSON 动作」类提示，而不是直接以该段文字作为任务完成总结。
