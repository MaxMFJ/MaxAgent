## 二、各功能详细说明

### 1. 代码级 Gather → Act → Verify 三阶段

**文件**: `backend/agent/exec_phases.py`

三阶段定义：
- **Gather**：信息收集动作（read_file, list_directory, think, get_system_info, web_search）
- **Act**：产生副作用的执行动作（run_shell, write_file, move_file, delete_file, call_tool 等）
- **Verify**：自动化验证（位于 Act 之后，代码自动执行）

实现方式：
1. `PhaseTracker` 类：附加到每个自主任务，记录每次迭代的阶段和结果
2. `infer_phase(action_type)` 函数：根据 action_type 判断所属阶段
3. `auto_verify(action_type, params, result)` 异步函数：执行轻量级验证检查：
   - `write_file` → 检查文件是否存在及大小
   - `delete_file` → 确认文件已不存在
   - `move_file` / `copy_file` → 确认目标路径存在
   - `run_shell` → 检查退出码是否为 0
4. 验证结果作为 `[Phase: VERIFY]` 消息注入 LLM 上下文
5. `task_complete` 事件中增加 `phase_stats` 字段（gather/act/verify 计数）

**新增 WebSocket 事件**:
```json
{"type": "phase_verify", "iteration": 3, "phase": "act", "note": "[Verify] write_file OK — 文件已存在，大小 1234 字节: /tmp/test.txt"}
```

### 2. MCP 生态集成

**文件**: `backend/agent/mcp_client.py` + `backend/routes/mcp.py`

支持两种传输协议：
- **stdio**：启动子进程，通过 stdin/stdout 做 JSON-RPC 通信（适用于 npx 启动的 npm 包形式 MCP server）
- **http**：通过 HTTP POST 做 JSON-RPC 通信

**API 端点**:
| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/mcp/servers` | 列出所有服务器及连接状态 |
| POST | `/mcp/servers` | 添加并连接 MCP 服务器 |
| DELETE | `/mcp/servers/{name}` | 断开并移除服务器 |
| GET | `/mcp/tools` | 列出所有工具 |
| POST | `/mcp/tools/call` | 调用工具 |

**使用示例**:
```bash
# 添加文件系统 MCP 服务器
curl -X POST http://localhost:8000/mcp/servers \
  -H 'Content-Type: application/json' \
  -d '{
    "name": "filesystem",
    "transport": "stdio",
    "command": ["npx", "-y", "@modelcontextprotocol/server-filesystem", "/Users/me/Documents"]
  }'

# 调用工具
curl -X POST http://localhost:8000/mcp/tools/call \
  -H 'Content-Type: application/json' \
  -d '{"server": "filesystem", "tool": "readFile", "arguments": {"path": "/Users/me/Documents/notes.txt"}}'
```

服务器配置持久化到 `backend/data/mcp_servers.json`，启动时自动重连。

#### 2.1 统一工具路由器（Unified Tool Router）

**核心设计**：`Agent → ToolRegistry → ToolRouter → Builtin Tools (优先) → MCP Adapter (fallback)`

解决的问题：MCP 工具原先仅通过 REST API 可用，Agent Core 执行任务时无法调用 MCP 工具。

**新增文件**:
- `backend/tools/mcp_adapter.py` — 核心适配层

**关键组件**:

1. **`MCPToolProxy(BaseTool)`** — 将 MCP 工具包装为标准 BaseTool：
   ```python
   class MCPToolProxy(BaseTool):
       # execute() 内部调用 MCPManager.call_tool(server, tool, args) 
       # 通过 JSON-RPC 与 MCP 进程通信
   ```

2. **智能命名策略**：
   | 场景 | 注册名 | LLM 可见 | 用途 |
   |------|--------|---------|------|
   | MCP 工具与内置重名 | `mcp/{server}/{tool}` | ❌ 隐藏 | Router fallback |
   | MCP 独有工具 | `{server}_{tool}` | ✅ 可见 | LLM 直接使用 |

3. **`sync_mcp_tools_to_registry(registry, mcp_manager)`** — 全量同步：
   - 启动时调用（`main.py` lifespan）
   - 动态调用：添加/移除 MCP 服务器时自动触发（`routes/mcp.py`）

4. **`find_mcp_fallback(registry, tool_name)`** — 查找隐藏的 MCP 备选

**修改文件**:

| 文件 | 修改 |
|------|------|
| `tools/router.py` | `execute_tool()` 失败后自动调用 `_try_mcp_fallback()` |
| `tools/registry.py` | `get_schemas()` 和 `get_relevant_schemas()` 过滤 `mcp/` 前缀工具 |
| `routes/mcp.py` | 添加 `_sync_mcp_to_registry()` 在增删服务器后同步 |
| `main.py` | lifespan 中 MCP 初始化后调用 `sync_mcp_tools_to_registry()` |

#### 2.2 Node.js 路径自动发现

**问题**：macOS 上通过 Homebrew `keg-only` 安装 Node.js 时（如 `brew install node@22`），`npx` 不在标准 PATH 中（`/opt/homebrew/bin/npx` 不存在），仅在 `/opt/homebrew/opt/node@22/bin/npx`。后端 Python 进程找不到 `npx`。

**解决**:

1. **`start.sh`** — 启动脚本自动发现：
   ```bash
   for _node_keg in /opt/homebrew/opt/node@*/bin /usr/local/opt/node@*/bin; do
       [ -d "$_node_keg" ] && _EXTRA_PATHS="$_EXTRA_PATHS $_node_keg"
   done
   ```

2. **`mcp_client.py` `_StdioTransport.start()`** — 运行时动态补充 PATH + `shutil.which()` 解析绝对路径

3. **错误诊断增强** — `_StdioTransport.call()` 捕获 `BrokenPipeError`/`TimeoutError`，读取 stderr 输出报告实际错误原因（如 "npm error 404 Not Found"）

### 3. 多模型路由（Fast / Strong / Cheap）

**文件**: `backend/agent/model_selector.py`

新增 `ModelTier` 枚举：
```python
class ModelTier(Enum):
    FAST   = "fast"    # 本地小模型 / 低延迟 API，用于简单操作
    STRONG = "strong"  # 旗舰远程模型，用于复杂推理
    CHEAP  = "cheap"   # 中等性价比远程模型，平衡质量与成本
```

**路由规则** (`get_tier_for_task`):
| 条件 | 路由至 |
|------|-------|
| 敏感数据 (privacy) | FAST (本地) |
| 复杂度 ≥ 7 / 复杂推理 / 多步规划 / 知识查询 | STRONG |
| 简单操作 | FAST |
| 其余 (代码生成等) | CHEAP |

**默认各级模型**:
| Tier | Provider | Model |
|------|----------|-------|
| FAST | local | 本地自动检测 |
| STRONG | deepseek | deepseek-reasoner |
| CHEAP | deepseek | deepseek-chat |

可通过 `llm_config.json` 中的 `tier_models` 字段覆盖：
```json
{
  "tier_models": {
    "strong": {"provider": "openai", "model": "gpt-4o"},
    "cheap": {"provider": "deepseek", "model": "deepseek-chat"}
  }
}
```

`model_selected` WebSocket 事件新增 `tier` 字段：
```json
{"type": "model_selected", "model_type": "remote", "tier": "strong", "reason": "..."}
```

### 4. 可回滚操作（文件快照 + Undo）

**文件**: `backend/agent/snapshot_manager.py` + `backend/routes/rollback.py`

**工作原理**：
1. 在 `write_file`、`delete_file`、`move_file`、`copy_file` 操作执行前，自动保存快照
2. 快照存储于 `~/.macagent/snapshots/<uuid>.snap.json`
3. 文件 ≤ 10 MB 时保存原始内容；超大或二进制文件只记录路径信息
4. 最多保留 500 条快照，自动淘汰最老的

**回滚逻辑**：
| 操作 | 回滚行为 |
|------|---------|
| write (新建) | 删除新建的文件 |
| write (覆盖) | 恢复原始内容 |
| delete | 恢复已删除文件 |
| move | 将文件移回原位置 |
| copy | 删除复制的文件 |

**API 端点**:
| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/rollback/snapshots` | 列出快照（支持 task_id / session_id 过滤）|
| POST | `/rollback` | 回滚指定快照 |
| DELETE | `/rollback/snapshots/{id}` | 删除快照记录 |

集成点：
- `tools/file_tool.py`（通过 `call_tool` 调用时）
- `agent/autonomous_agent.py`（直接 handle 时）：`_handle_write_file`, `_handle_move_file`, `_handle_copy_file`，以及 delete 中的 safety check 路径
