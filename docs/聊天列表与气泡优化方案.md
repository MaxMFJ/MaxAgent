# 聊天列表与气泡优化方案

## 一、当前问题

1. **AI 输入时闪烁、布局拉扯**
   - 流式响应每个 chunk 都调用 `updateAssistantMessage` → `updateCurrentConversation` → 替换整个 `currentConversation`，触发 `@Published`。
   - SwiftUI 整棵视图树重算；消息列表用 `VStack`，所有气泡参与布局，最后一个气泡内容频繁变化导致整表 layout。
   - `onChange(of: messages.count)` 等处的 `withAnimation` 与流式的高频更新叠加，加重闪烁。
   - Markdown 每条消息的 `content` 变化都会重新解析、重新布局，流式时频率过高。

2. **功能扩展性不足**
   - 气泡内容以「一条 Message + Markdown + 附件」为主，若要支持多种类型（工具调用、代码块、结构化卡片等）需要可扩展的气泡内容结构。

---

## 二、是否接入 IGListKit？

**结论：不建议接入。**

| 维度 | 说明 |
|------|------|
| 平台 | [IGListKit](https://github.com/Instagram/IGListKit) 基于 **UIKit 的 UICollectionView**，面向 iOS/tvOS。 |
| macOS | 官方仅支持 macOS 10.13+ 的 **diffing 算法组件**，不提供完整 NSCollectionView 适配。 |
| 技术栈 | 当前 Mac 端为 **SwiftUI**，接入需用 `UIViewRepresentable` 包一层 UICollectionView，且 Mac 上通常用 NSCollectionView，与 IGListKit 的 API 不对齐。 |
| 成本 | 需重写整个聊天列表为 UIKit/AppKit，与现有 SwiftUI 风格不一致，维护成本高。 |

在 SwiftUI 内用「节流 + 懒加载 + 稳定 identity + 可扩展气泡模型」即可达到类似效果，且与现有架构一致。

---

## 三、SwiftUI 方案概览

### 3.1 消除闪烁与布局拉扯

1. **流式更新节流（Throttle）**
   - 在 `updateAssistantMessage` 中，当 `isStreaming == true` 时，对写库/触发 UI 的更新做节流（例如每 80–100ms 最多一次），减少 `@Published currentConversation` 的触发频率。
   - 节流仅针对「写回 conversation」的时机；后端仍可照常逐 chunk 推送，最后一条消息的「当前展示内容」由节流后的更新驱动。

2. **列表懒加载**
   - 将 `ScrollView` 内 `VStack` 改为 **LazyVStack**，只对可见（及附近）消息做布局与 body 计算，流式时仅最后一条气泡频繁变化，其它行不参与布局。

3. **滚动与动画**
   - 流式期间避免因「内容变化」而触发的 `scrollTo` 动画；仅在「新消息出现」或「流式结束」时再滚动到底部，并控制动画时长，减少与内容更新的叠加。

4. **（可选）流式缓冲与 conversation 解耦**
   - 若仍觉抖动明显，可增加「当前流式缓冲」状态（如 `@Published var streamingContent: String?`），仅最后一条气泡在 `isStreaming` 时绑定该缓冲；流式过程中不修改 `conversation.messages`，结束时再写回。这样列表的「数据源」在流式期间不变，仅单条气泡视图更新。

### 3.2 可扩展的气泡内容（不依赖 IGListKit）

目标：支持多种气泡内容类型（纯文本/Markdown、附件、工具调用、代码块、后续扩展类型），且便于新增类型。

- **方案 A：基于 Message 的 content + 附件 + 扩展字段**
  - 保持现有 `Message` 结构，通过 `MessageRole`、`attachments`、或新增 `messageKind: MessageKind` 枚举（如 `.text`, `.toolCall`, `.card`）区分展示。
  - 在 `MessageBubble` / `AssistantMessageContent` 内用 `@ViewBuilder` + `switch messageKind` 渲染不同 UI，新类型只需扩展枚举和对应 view。

- **方案 B：消息「块」列表**
  - 将一条消息拆成多个「块」（如文本块、图片块、工具调用块），模型为 `MessageBlock: Identifiable`，`Message` 持有 `blocks: [MessageBlock]`。
  - 气泡内用 `ForEach(message.blocks)` 按块类型渲染，扩展时新增 block 类型即可。

建议先做 **方案 A**（枚举 + ViewBuilder），实现简单、与现有 `Message` 兼容；若后续需要更细粒度（如一条消息内多种块混排），再引入 **方案 B**。

---

## 四、实施顺序

1. **先做**：流式节流 + LazyVStack + 滚动/动画收敛（立即缓解闪烁与拉扯）。
2. **再做**：可扩展气泡内容（MessageKind 或 MessageBlock），便于后续加工具调用、卡片等。
3. **视情况**：若仍有个别机型/场景抖动，再考虑「流式缓冲与 conversation 解耦」。

---

## 五、涉及文件

| 文件 | 修改点 |
|------|--------|
| `AgentViewModel.swift` | 流式更新节流（`updateAssistantMessage` 内，isStreaming 时限频写回 conversation）。 |
| `ChatView.swift` | VStack → LazyVStack；滚动逻辑仅在新消息/流式结束时触发，避免与节流更新叠加。 |
| `Message.swift` | 可选：新增 `MessageKind` 或扩展字段，为可扩展气泡做准备。 |
| `MessageBubble.swift` / `AssistantMessageContent.swift` | 按 MessageKind 或 blocks 分支渲染，支持多种气泡内容。 |

上述方案不依赖 IGListKit，全部在现有 SwiftUI 架构内完成，便于维护和扩展。
