# v3.1 跟进计划（2025–2026 研究/社区共识）

基于常见失败模式与社区实践，按优先级列出优化方向与代码级落地思路。v3.1 在 v3 框架上做「上下文 / 规划 / 升级 / 反思 / 安全 / 目标对齐」增强，不破坏现有主流程。

**实施状态**：v3.1 已落地。见 `app_state` 中 FeatureFlag、`agent/safety.py`、`autonomous_agent` 中 summarize/Goal/Plan/Reflection/Escalation。

---

## 优先级总览

| 优先级 | 方向 | 当前痛点 | 预期收益 |
|--------|------|----------|----------|
| ★★★★★ | 结构化 memory + 上下文压缩 | 字符串拼接导致 context 污染、token 爆炸、关键历史被稀释 | 显著降低 long-horizon 漂移，token 可控 |
| ★★★★☆ | Plan-and-Execute 分离 | 纯 ReAct 单步易陷局部最优，缺全局规划 | 长任务成功率提升 10–30% |
| ★★★★ | Escalation 智能触发 | 硬编码 2/3 次相同 hash，误触发或延迟触发 | 更智能 escalation，减少无效循环 |
| ★★★☆ | 反思时机与模型 | 仅结束/停止时反思，且仅本地模型 | 中途纠错、高质量反思降低早停 |
| ★★★ | 危险命令/工具统一校验 | 仅 run_shell 有 _is_dangerous_command，call_tool 无统一 audit | 安全与审计能力提升 |
| ★★☆ | Goal 重述与对齐 | 长循环易偏离原始 task | 简单有效防 drift |

## 1. ★★★★★ 结构化 memory 与上下文压缩

### 问题描述

- 当前上下文构建依赖 **字符串拼接**：`TaskContext.get_context_for_llm()`（`action_schema.py`）产出纯文本，只取最近 10 条 action_log，每条 reasoning/output/error 截断到 50/200/100 字。
- 手动在 `_generate_action` 里注入 escalation 文本（`_escalation_prompt`），与上面拼在一起。
- 长时序任务中 context 逐渐污染、token 爆炸、关键历史被稀释 → reasoning 漂移或 hallucinated actions。

### 常见失败模式

- 迭代 20+ 后模型“忘记”早期成功步骤，重复失败动作。
- token 超限导致截断，关键错误信息被裁掉。
- 无法按“重要性”保留历史（例如最后一次失败 vs 早期一次成功）。

### 建议优化方向（代码级）

- **结构化 memory**：用 `list[dict]` 存每步 `{iteration, thought, action_type, action_params_summary, observation, success}`，不替代现有 `action_logs`，可与其并存或逐步迁移。
- **动态 summarize/compress**：
  - 规则：最近 N 条完整保留，更早的做“摘要”（如每 5 步合并为一句：`步骤 1–5：完成了 X，失败 Y`）。
  - 可选：小模型或 rule-based 生成摘要，再拼进 prompt。
- **Prompt 构建**：由“整段字符串”改为“结构化条目 + 摘要段 + 当前轮 escalation”，总长度上限可配置（如 4000 chars 或 2048 tokens 估算）。

### 涉及文件与接口

| 位置 | 建议 |
|------|------|
| `agent/action_schema.py` | `TaskContext` 增加 `structured_steps: List[dict]` 或新方法 `get_structured_history()`；新增 `summarize_history_for_llm(max_chars, recent_n)` 返回 (recent_text, summary_text)。 |
| `agent/autonomous_agent.py` | `_generate_action` 中不再直接调用 `context.get_context_for_llm()`，改为调用上述 summarize 接口，再拼接 escalation；或先保留旧逻辑，新逻辑用 FeatureFlag 开关。 |
| 可选新模块 | `core/task_memory.py` 或 `agent/task_memory.py`：定义 `StepRecord`、`compress_steps(steps, max_tokens_approx)`，供多处复用。 |

### 预期收益

- 显著降低 long-horizon 漂移；token 可配置、可观测。

## 2. ★★★★☆ Plan-and-Execute 分离

### 问题描述

- 当前为纯 ReAct 风格：单步生成 → 执行 → 观察 → 再生成，无显式“先规划再执行”。
- 复杂任务易陷局部最优（反复小步失败），缺乏全局规划。

### 建议优化方向（代码级）

- **可选的 planning phase**：任务开始时（或 FeatureFlag 开启时）先调用一次 LLM，产出 high-level plan：`sub_tasks: List[str]`。
- **执行循环**：按 sub_tasks 逐个执行（或当前 step 指向某 sub_task）；允许执行中“replan”（如连续失败 2 次则重新生成剩余 sub_tasks）。
- 参考：Plan-and-Execute、Reflexion 等：plan 与 execute 分离，必要时 replan。

### 涉及文件与接口

| 位置 | 建议 |
|------|------|
| `agent/autonomous_agent.py` | 在 `run_autonomous` 开头增加可选分支：若启用 plan，则先 `_generate_plan(task)` 得到 `plan: List[str]`，再进入现有 while 循环；循环内可带 `current_plan_index` 与 replan 触发条件。 |
| `agent/action_schema.py` 或新 schema | 定义 `Plan`（如 `sub_tasks: List[str], created_at`），便于持久化与注入 prompt。 |

### 预期收益

- 长任务成功率提升 10–30%（社区常见量级）；需评估单次 plan 的 token 成本。

## 3. ★★★★ Escalation 与 skill fallback 智能触发

### 问题描述

- 当前 `_detect_repeated_failure`（`autonomous_agent.py`）用 **md5(type + output[:200])** 判断“相同”失败，连续 2 次同 hash → FORCE_SWITCH，3 次 → SKILL_FALLBACK。
- 误触发：语义相近但输出略有不同的失败被当成不同，未升级；或不同失败因截断后 hash 相同被误判为重复。
- 延迟触发：必须严格“相同”才计数，导致无效循环多。

### 建议优化方向（代码级）

- **相似度替代 hash**：用 BGE-small 等对“(action_type, error/observation)”做 embedding，用余弦相似度判断“是否重复失败”；阈值可配置（如 >0.85 视为同质失败）。
- **失败类型分类**：对每次失败打标 `failure_type: tool_error | parse_error | logic_error`，escalation 策略可按类型区分（如 tool_error 优先换工具，logic_error 触发 skill 提示）。
- **阈值可配置**：连续 N 次“相似失败”触发 FORCE_SWITCH / SKILL_FALLBACK，N 从配置或 FeatureFlag 读，避免硬编码 2/3。

### 涉及文件与接口

| 位置 | 建议 |
|------|------|
| `agent/autonomous_agent.py` | `_detect_repeated_failure` 改为基于 embedding 相似度 + 配置阈值；可调用现有 `agent/vector_store` 或单独小模块算相似度。 |
| `config` 或 `app_state` | 增加 `escalation_similarity_threshold`、`escalation_force_after_n`、`escalation_skill_after_n`。 |
| 可选 | 在 `core/error_model.py` 中为 `AgentError` 或失败记录增加 `failure_type`，供 escalation 使用。 |

### 预期收益

- 更智能的 escalation，减少误触发与无效循环。

## 4. ★★★☆ 反思时机与模型

### 问题描述

- 当前 reflection 仅在 **任务结束/停止** 时跑（`run_autonomous` 里在 yield task_complete / task_stopped 之后），且只用 **本地模型**（如 Ollama）。
- 反思晚 → 无法中途纠错；本地模型弱 → 反思质量低，consecutive_failures 仍易导致早停。

### 建议优化方向（代码级）

- **Mid-loop reflection**：每 N 步（如 5）或连续失败 K 次时触发一次“轻量反思”，输出简短建议并回写到 context（作为 memory update 或一条 system/user 消息）。
- **Remote 反思开关**：允许用 remote 模型做高质量反思（配置或 FeatureFlag），并限制频率以控制 token 成本。
- **反思结果回写**：反思结果写入 `TaskContext` 或 structured memory，下一轮 `get_context_for_llm` / summarize 时包含，避免“反思了但模型看不到”。

### 涉及文件与接口

| 位置 | 建议 |
|------|------|
| `agent/autonomous_agent.py` | 在主循环内增加条件：`if (iteration % 5 == 0 or consecutive_failures >= 2) and enable_mid_reflection` 则调用 `_run_reflection` 的轻量版，将结果注入后续 prompt。 |
| `agent/reflect_engine.py` 或反思入口 | 支持“轻量模式”（更短 prompt、只输出建议列表）；支持指定 model（local/remote）。 |
| `TaskContext` / memory | 增加字段或步骤类型 `reflection_summary`，在 summarize 时一并包含。 |

### 预期收益

- 显著降低因 consecutive_failures 导致的早停；中途纠错提升长任务表现。

## 5. ★★★ 危险命令/工具统一校验与审计

### 问题描述

- 当前仅 `_handle_run_shell` 中有 `_is_dangerous_command(command)`（`autonomous_agent.py`）；`call_tool` 等路径无统一校验，custom tool 可能执行高危操作（如写 shell、删系统文件）。
- 无统一 audit trail：谁在何时执行了何种 action，仅在内存中，finish 前崩溃则丢失。

### 建议优化方向（代码级）

- **统一 pre-execute validator**：在 `_execute_action` 入口（或每个 handler 调用前）做一次统一校验：`dangerous_patterns`（命令/路径/参数）+ 简单 permission check（如禁止写 /System）。
- **call_tool 参数校验**：对 `tool_name` + `args` 做白名单/危险模式扫描（可与现有 `capsule_validator` 危险命令逻辑复用）。
- **Action log 持久化**：每步执行后立即 append 到 `data/executions/{task_id}.json` 或类似（与 v3 执行轨迹持久化一致），即使未 finish 也可事后排查。

### 涉及文件与接口

| 位置 | 建议 |
|------|------|
| `agent/autonomous_agent.py` | `_execute_action` 开头调用 `validate_action_safe(action)`；将 `_is_dangerous_command` 抽成通用 `core/safety.py` 或 `agent/safety.py`，并扩展 `dangerous_patterns`、路径黑名单；`call_tool` 分支内对 args 做同样校验。 |
| 执行轨迹 | 与 v3 执行轨迹持久化对接：每步 append，不只在 task_complete 时写。 |

### 预期收益

- 安全与审计能力提升；便于事后 debug。

## 6. ★★☆ Goal 重述与对齐

### 问题描述

- 长循环中模型逐渐偏离原始 task（drift），缺少显式的 goal 再确认。

### 建议优化方向（代码级）

- **周期性重述**：每 5–8 步在 prompt 中强制插入一句：`【当前目标】原始任务: {task}\n 当前应完成: ...`（可由上一步结果简要推导）。
- **可选 alignment score**：用小模型或规则对“当前计划/动作”与“原始 task”做一致性打分，低于阈值时注入更强提醒。

### 涉及文件与接口

| 位置 | 建议 |
|------|------|
| `agent/autonomous_agent.py` | 在 `_generate_action` 构建 messages 时，若 `context.current_iteration % 6 == 0`（可配置）则在 user 内容前插入一行 goal 重述。 |
| `agent/action_schema.py` | `get_context_for_llm` 或新的 summarize 接口可在开头固定带一句 `任务: {task_description}`，保证至少出现一次。 |

### 预期收益

- 简单有效防 drift（Ralph Loop 等 2026 实践常用）。

## 实施顺序建议

1. **Phase A（高收益、可渐进）**  
   - 结构化 memory + summarize 接口（与现有 `get_context_for_llm` 并存，FeatureFlag 切换）。  
   - Goal 重述：每 N 步注入原始 task（改动小）。

2. **Phase B**  
   - Escalation：embedding 相似度 + 配置阈值 + failure_type。  
   - 统一 safety validator + action log 持久化。

3. **Phase C**  
   - Mid-loop reflection + 可选 remote 反思。  
   - 可选 Plan-and-Execute 阶段与 replan。

## 与 v3 的关系

- v3 已提供：错误模型、状态机、并发限流、TimeoutPolicy、config 合并。  
- v3.1 在此基础上增加：**上下文与 memory 结构、规划/反思/安全/目标对齐**，不破坏 v3 接口；新逻辑尽量通过 FeatureFlag 或配置开关，便于回滚与 A/B。
