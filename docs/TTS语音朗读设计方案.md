# TTS / STT 语音方案（朗读 + 识别）

> 目的：明确 TTS/STT 放在后端还是客户端；**语音识别（STT）→ 文字 → 现有 LLM 对话**在客户端实现即可形成「能听能说」闭环，无需改后端。

---

## 一、现状

- **notification 工具**：已有 `action: "speak"`，调用 `runtime_adapter.speak(message)`。
- **Mac 后端**：`mac_adapter.speak()` 使用系统命令 `say` 在**运行后端的机器**上朗读。
- **多端**：Mac/iOS 可同时连接同一后端；若后端在 Mac 上，`say` 只在 Mac 发声，iOS 端听不到。
- **流式回复**：后端已通过 WebSocket 推送 `type: "content"` 的文本流，客户端已有完整回复文本。

---

## 二、两种 TTS 场景

| 场景 | 说明 | 谁需要发声 |
|------|------|------------|
| **A. 朗读助手回复** | 用户勾选「朗读回复」或点击「朗读」按钮，把当前/流式助手内容读出来 | 当前使用中的设备（Mac 或 iPhone） |
| **B. 显式朗读指令** | 用户说「用语音读一下这段」或 Agent 调用 notification(speak) 主动朗读某段内容 | 同上，应是「正在对话的那台设备」 |

结论：两种场景都应以**当前会话所在客户端**为发声端，而不是后端所在机器。

---

## 三、后端 TTS vs 客户端 TTS

### 方案对比

| 维度 | 后端 TTS | 客户端 TTS |
|------|----------|-------------|
| **发声位置** | 后端机器（如 Mac 跑 say/API） | 各客户端本机（用户面前的设备） |
| **多端** | 仅后端所在设备有声，iOS 等听不到 | 谁在会话谁发声，自然支持多端 |
| **延迟** | 需先合成再传音频流，首包延迟更高 | 拿到文本即可读，延迟低 |
| **带宽** | 需传音频流，占用大 | 只传文本（已有），不额外占带宽 |
| **离线** | 依赖后端与（若用云 API）网络 | 系统 TTS 可完全离线 |
| **音色统一** | 可用同一云 TTS，多端一致 | 各端系统语音，音色可能不同 |
| **实现成本** | 后端需集成 TTS API 或本地引擎，新消息类型、流式音频 | 客户端用系统 API（如 AVSpeechSynthesizer）即可 |

### 推荐结论

- **默认推荐：以客户端 TTS 为主**
  - 场景 A（朗读助手回复）：客户端已有流式/完整文本，直接交给系统 TTS 朗读即可。
  - 场景 B（显式朗读）：后端不直接执行 `say`，改为向当前会话推送一条「朗读指令」（如 WebSocket `type: "speak"`），由**收到该消息的客户端**在本机执行 TTS。

- **何时考虑后端 TTS**
  - 需要**跨设备统一音色**（如统一品牌声线）时，可再增加「后端合成 → 推送音频流」的可选通道。
  - 或后端作为无头服务、没有扬声器的环境，且希望由某台指定设备播放时，仍可通过「后端发 speak 指令 → 指定客户端播放」解决，无需后端自己合成。

## 四、推荐架构（客户端 TTS + 后端仅发「朗读指令」）

### 4.1 场景 A：朗读助手回复

- **数据**：不新增协议。继续用现有流式 `content` 与 `done`，客户端已有完整回复文本。
- **逻辑**：
  - 设置项增加「朗读助手回复」开关（可默认关）。
  - 流式过程中：客户端按句或按段（如遇到 `。！？\n`）将文本送入本机 TTS 队列朗读；或等 `done` 后整段朗读。
  - 若用户取消/切换会话，停止当前朗读。
- **实现位置**：仅客户端（Mac：`AVSpeechSynthesizer` / `NSSpeechSynthesizer`，iOS：`AVSpeechSynthesizer`）。

### 4.2 场景 B：显式朗读（含 notification(speak)）

- **现状问题**：当前 `notification(speak)` 在后端执行 `runtime_adapter.speak()`，在 Mac 上即本机 `say`，多端时只有后端所在设备有声。
- **调整**：
  1. **后端**：不再在服务端调用 `runtime_adapter.speak()`（或仅保留为「本机朗读」的可选行为）。  
     当 Agent 调用 notification(speak, message) 时，后端向**当前会话**推送一条 WebSocket 消息，例如：
     ```json
     { "type": "speak", "text": "要朗读的完整内容", "request_id": "可选去重" }
     ```
  2. **客户端**：收到 `type: "speak"` 后，用本机 TTS 朗读 `text`；可支持队列（多条 speak 顺序读）或打断当前朗读。
- **兼容**：若需保留「后端本机也读」的行为（例如后端跑在用户主 Mac 上），可配置为：后端在推送 `speak` 给客户端的同时，仍调用 `runtime_adapter.speak()`。

### 4.3 协议约定（仅新增 speak 事件）

- **后端 → 客户端**（WebSocket）  
  - `type: "speak"`  
  - `text: string`：要朗读的文本  
  - 可选：`request_id`、`interrupt: bool`（是否打断当前朗读）

- **客户端**  
  - 在现有 `StreamChunk` / 消息分发处增加对 `speak` 的处理，调用本地 TTS 即可。

## 五、实现清单（按推荐方案）

### 后端

- [ ] **notification_tool**：在 `action == "speak"` 时，不直接依赖 `runtime_adapter.speak()` 作为唯一路径；改为通过「当前会话」下发 speak 事件（需要 ws_handler 或 connection_manager 支持「向 session 广播 speak」）。
- [ ] **ws_handler / connection_manager**：提供「向某 session 广播 `{ type: "speak", text }`」的能力；若 session 对应多个连接（多端），则多端都会收到，由各端本机朗读。
- [ ] （可选）保留 `runtime_adapter.speak()` 的调用，作为「后端本机也读」的可选逻辑，由配置或环境决定是否执行。

### Mac 客户端

- [ ] **设置**：增加「朗读助手回复」开关；可选「仅完整回复后朗读」或「流式按句朗读」。
- [ ] **TTS 服务**：封装 `AVSpeechSynthesizer` 或 `NSSpeechSynthesizer`，支持：播报、暂停/继续、停止、按句排队。
- [ ] **流式/完成回调**：在收到 `content` 或 `done` 时，若开启朗读，把文本送入 TTS 服务。
- [ ] **WebSocket**：在解析到 `type: "speak"` 时，将 `text` 送入同一 TTS 服务朗读。

### iOS 客户端

- [ ] 同上：设置项、TTS 封装（`AVSpeechSynthesizer`）、流式/完成时朗读、处理 `speak` 消息。

## 六、小结

| 问题 | 结论 |
|------|------|
| TTS 放在后端还是客户端？ | **以客户端 TTS 为主**，朗读在用户当前使用的设备上执行。 |
| 助手回复朗读？ | 客户端用现有流式/完整文本 + 系统 TTS，不新增后端协议。 |
| notification(speak) 多端如何一致？ | 后端只发 `type: "speak"`，由各端客户端本机朗读；可选保留后端本机 `say`。 |
| 是否需要后端合成音频？ | 仅在需要「统一音色」或特殊需求时再考虑后端 TTS + 音频流。 |

按上述设计，后端只需增加「speak 事件推送」和 notification 工具的小改，主要开发在客户端的 TTS 封装与设置/流式联动即可。

## 七、语音识别（STT）与「能听能说」闭环

### 7.1 闭环流程

```
用户说话 → 客户端麦克风 → 客户端 STT（语音转文字）→ 文字作为 chat 内容发送后端 → LLM 流式回复
         → 可选：客户端 TTS 朗读回复
```

**结论：在客户端实现「语音识别 → 转文字 → 走现有 chat 接口」即可闭环，后端无需改协议。**  
后端已经接收 `type: "chat"`, `content: "文本"`，不关心内容来自键盘还是 STT，所以 STT 完全在客户端完成即可。

### 7.2 为何 STT 也放客户端？

| 维度 | 客户端 STT | 后端 STT |
|------|------------|----------|
| **麦克风** | 本机录音，天然在客户端 | 需把音频流上传，占带宽、延迟高 |
| **隐私** | 可用系统 on-device 识别，语音不出设备 | 音频上传后端，需考虑隐私与合规 |
| **协议** | 只上传「识别结果文本」，复用现有 chat | 需新协议传音频或音频 URL |
| **实现** | 系统 API：macOS `SFSpeechRecognizer`、iOS `SFSpeechRecognizer` / 听写 | 需后端集成语音识别服务 |

与 TTS 一致：**谁在交互谁采集、谁输出**，客户端负责「听」和「说」，后端只处理「文本对话」。

### 7.3 实现要点（仅客户端）

- **麦克风权限**：请求录音权限（Mac/iOS 需在 Info 中声明用途）。
- **语音识别**：  
  - **Mac**：`Speech.framework`（`SFSpeechRecognizer` + `SFSpeechAudioBufferRecognitionRequest`），支持连续识别、实时结果。  
  - **iOS**：同上；或短句场景用「按住说话」+ 结束送识别的简化流程。
- **结果处理**：识别得到最终/中间文本后，填入输入框或直接调用现有「发送消息」逻辑（与键盘发送同 path），即 `sendMessageStream(识别文本, sessionId)`。
- **可选**：UI 上提供「语音输入」按钮（点击开始录音 → 再点或自动静音检测结束 → 送识别结果发往 LLM）；若开启「朗读回复」，则整条链路为：**说 → STT → LLM → 流式回复 → TTS 读**。

### 7.4 STT 实现清单（客户端）

- [ ] **Mac 客户端**：Speech 权限、`SFSpeechRecognizer` 封装（含语言/中文）、录音 buffer 送入识别、结果回调后调用现有发送消息接口。
- [ ] **iOS 客户端**：同上；注意后台/权限与短句 UI（如按住说话）。
- [ ] **后端**：无变更；继续接收 `chat` 文本即可。

---

## 八、小结（含 STT）

| 问题 | 结论 |
|------|------|
| TTS 在哪实现？ | **客户端**，用系统 TTS 读回复 / 执行 speak 指令。 |
| STT 在哪实现？ | **客户端**，语音 → 文字后走现有 chat 接口，**流程即闭环**。 |
| 后端要改吗？ | TTS：仅需增加 `speak` 事件推送。STT：**不需要**，只收文本。 |
| 闭环是什么？ | 用户说话 → 客户端 STT → 文本发 LLM → 流式回复 → 可选客户端 TTS 朗读 = 能听能说。 |
