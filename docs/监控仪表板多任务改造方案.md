# 监控仪表板多任务改造方案

## 问题分析

当前监控仪表板存在**单线监控**限制：

1. **数据模型**：`MonitoringViewModel` 仅维护单一 `actionLogs`、`taskProgress`、`llmStreamingText`
2. **事件处理**：`AgentViewModel.handleMonitorEvent` 接收所有任务的 `monitor_event`，但**不按 task_id 区分**，直接合并到同一数组
3. **结果**：多任务并行时（主任务 + SubAgent、多会话 Chat、多窗口）：
   - `task_start` 互相覆盖
   - `action_plan` / `action_result` 混入同一时间轴
   - `action_id` 可能跨任务冲突
   - 无法直观看到「哪个任务在跑」「每个任务进度如何」

## 解决方案概览

| 层级 | 改造内容 |
|------|----------|
| **后端** | 新增 `GET /monitor/active-tasks` 返回运行中任务列表；确保 `action_id` 含 task_id 前缀 |
| **Mac ViewModel** | 按 task_id 分桶存储：`tasks: [task_id: TaskMonitorData]` |
| **Mac UI** | 左侧任务列表/标签页 + 右侧选中任务详情；顶部可加「全部任务概览」 |

---

## 一、后端改造

### 1.1 新增 API：`GET /monitor/active-tasks`

```python
# routes/monitor.py
@router.get("/active-tasks")
async def get_active_tasks():
    """返回当前运行中及最近完成的任务列表，供监控面板多任务展示"""
    tracker = get_task_tracker()
    tasks = []
    for task_id, tt in tracker._tasks.items():
        if tt.status == AutoTaskStatus.RUNNING or (tt.finished_at and (time.time() - tt.finished_at) < 300):
            tasks.append({
                "task_id": task_id,
                "session_id": tt.session_id,
                "task_type": tt.task_type.value,
                "description": tt.task_description,
                "status": tt.status.value,
                "started_at": tt.started_at,
                "finished_at": tt.finished_at,
            })
    return {"tasks": tasks}
```

### 1.2 确保 action_id 全局唯一（可选，建议）

在 `autonomous_agent.py` 发送 `action_plan` 时，将 `action_id` 改为 `{task_id}_{action_id}`，避免跨任务冲突。当前 `action_id` 来自 LLM/本地生成，若已含 UUID 则冲突概率低，可先不改，观察后再定。

## 二、Mac 端数据模型改造

### 2.1 新增 `TaskMonitorData`

```swift
// MonitoringViewModel 或独立文件
struct TaskMonitorData: Identifiable {
    let id: String           // task_id
    var taskProgress: TaskProgress?
    var actionLogs: [ActionLogEntry]
    var llmStreamingText: String
    var isStreamingLLM: Bool
    var sessionTokenUsage: TokenUsage
    var currentIteration: Int
    var selectedModelType: String?
    var selectedModelReason: String?
    var taskComplexity: Int
    var taskElapsedSeconds: Int
    var sourceSession: String
    var taskType: String     // "chat" | "autonomous"
    var lastUpdated: Date

    init(taskId: String, sourceSession: String, taskType: String) {
        self.id = taskId
        self.taskProgress = nil
        self.actionLogs = []
        self.llmStreamingText = ""
        self.isStreamingLLM = false
        self.sessionTokenUsage = TokenUsage()
        self.currentIteration = 0
        self.selectedModelType = nil
        self.selectedModelReason = nil
        self.taskComplexity = 0
        self.taskElapsedSeconds = 0
        self.sourceSession = sourceSession
        self.taskType = taskType
        self.lastUpdated = Date()
    }
}
```

### 2.2 MonitoringViewModel 改造

```swift
// 替换原来的单一数据
@Published var tasks: [String: TaskMonitorData] = [:]  // task_id -> data
@Published var selectedTaskId: String? = nil
@Published var activeTaskIds: [String] = []  // 用于 UI 列表排序（运行中优先）

// 兼容旧逻辑：selectedTaskId 对应的数据，或「当前会话」任务
var currentTaskData: TaskMonitorData? {
    if let id = selectedTaskId, let t = tasks[id] { return t }
    return tasks.values.first { $0.taskProgress?.status == .running }
        ?? tasks.values.max(by: { ($0.lastUpdated) < ($1.lastUpdated) })
}
```

### 2.3 AgentViewModel.handleMonitorEvent 改造

**关键**：按 `taskId` 路由到对应 `TaskMonitorData`，不再写入单一 `actionLogs`。

```swift
private func handleMonitorEvent(sourceSession: String, taskId: String, event: [String: Any]) {
    // 将事件转发给 MonitoringViewModel，由它按 taskId 分桶
    monitoringViewModel?.applyMonitorEvent(taskId: taskId, sourceSession: sourceSession, event: event)
}
```

`MonitoringViewModel.applyMonitorEvent` 内部逻辑：

- 若 `tasks[taskId]` 不存在，则创建并加入 `tasks`
- 根据 `event["type"]` 更新对应 `TaskMonitorData` 的 `actionLogs`、`taskProgress` 等
- 更新 `activeTaskIds`（运行中在前，按时间排序）
- `action_id` 查找时限定在 `tasks[taskId].actionLogs` 内

## 三、Mac UI 改造

### 3.1 ExecutionTimelineView 布局

```
┌─────────────────────────────────────────────────────────────────┐
│  [任务列表]          │  [选中任务详情]                             │
│  ─────────────       │  ─────────────                             │
│  ● 任务A (运行中)     │  TaskHeaderBar + TimelineScrollView        │
│  ○ 任务B (已完成)    │  + NeuralStreamPanel + TimelineSidebar    │
│  ○ 任务C (已完成)    │                                            │
│  ─────────────       │                                            │
│  [刷新] [全部]        │                                            │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 任务列表组件

- 左侧 `List` 或 `VStack` 展示 `activeTaskIds` 对应的任务
- 每行：状态图标 + 任务描述（截断）+ 步数/耗时
- 点击选中，`selectedTaskId = taskId`
- 支持「全部」视图：合并展示所有任务的时间轴（按时间排序，每条带 task_id 标签）

### 3.3 数据流

- `ExecutionTimelineView` 读取 `vm.currentTaskData` 或 `vm.tasks[vm.selectedTaskId ?? ""]`
- `TimelineScrollView`、`TimelineSidebar` 改为接收 `TaskMonitorData` 而非直接读 `vm.actionLogs`

## 四、实施步骤建议

1. **Phase 1（最小改动）**  
   - 后端：新增 `GET /monitor/active-tasks`  
   - Mac：`MonitoringViewModel` 增加 `tasks` 字典，`handleMonitorEvent` 按 task_id 分桶  
   - UI：左侧加任务列表，点击切换 `selectedTaskId`，右侧展示选中任务

2. **Phase 2（体验优化）**  
   - 轮询 `/monitor/active-tasks` 刷新任务列表  
   - 「全部」视图：合并多任务时间轴，每条带任务标签  
   - 新任务自动选中、运行中任务高亮

3. **Phase 3（可选）**  
   - Web 端 `MonitorDashboard` 同步改造  
   - SubAgent 父子关系在 UI 中展示（需后端在 event 中带 `parent_task_id`）

---

## 五、与现有组件的衔接

- `MonitoringViewModel.subscribeToAgentViewModel`：可保留，用于**当前会话**的实时数据（若 AgentViewModel 仍维护当前会话的 actionLogs，可双写一份到 `tasks[sessionTaskId]`）；或改为完全由 `monitor_event` 驱动，不再订阅 AgentViewModel 的执行数据。
- `BackendService.onMonitorEvent`：已收到所有任务的 event，只需在 `handleMonitorEvent` 中按 task_id 分桶即可。
- `TracesDashboardView`：已有 trace 按 task_id 区分，可复用 `/traces` 数据做历史任务入口。

---

## 六、注意事项

1. **会话与任务的对应**：同一 session 可能先后有多个 task（如连续两次 autonomous），需以 task_id 为准。
2. **内存控制**：`tasks` 可限制最多保留 N 个已完成任务（如 20 个），超出的从字典移除。
3. **Web 端**：若 `web/` 下有 MonitorDashboard，需同样改为多任务数据结构。
