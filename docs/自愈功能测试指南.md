# 自愈功能测试指南

> 后端默认端口 `8765`，请先启动后端再测试。若使用隧道/代理，将 `localhost:8765` 换成实际 base URL。

---

## 一、HTTP API 测试（curl / Postman）

### 1. 查询自愈状态

```bash
curl -s http://localhost:8765/self-healing/status | jq
```

**预期**：返回 `current_status`、`statistics`、`recent_healings`。

---

### 2. 仅诊断（不执行修复）

模拟一条工具解析失败错误，看诊断引擎能否正确分类：

```bash
curl -s -X POST http://localhost:8765/self-healing/diagnose \
  -H "Content-Type: application/json" \
  -d '{
    "error_message": "No tool call found in LLM response",
    "stack_trace": "  File \"agent/local_tool_parser.py\", line 42\n    raise ParseError(...)",
    "context": {"session_id": "test-session"}
  }' | jq
```

**预期**：返回 `problem_type`（如 `tool_parse_failure`）、`severity`、`description`、`suggestions`。

可替换的典型错误消息（用于覆盖不同问题类型）：

| 问题类型 | error_message 示例 |
|----------|--------------------|
| 工具解析失败 | `No tool call found`、`Failed to parse tool` |
| LLM 空响应 | `Empty response from LLM` |
| 连接错误 | `Connection refused. Ollama not running` |
| 超时 | `Request timed out after 90s` |
| 上下文丢失 | `无法看到之前的对话` |
| 权限 | `Permission denied`、`EACCES` |

---

### 3. 诊断 + 修复计划（不执行）

```bash
curl -s -X POST http://localhost:8765/self-healing/plan \
  -H "Content-Type: application/json" \
  -d '{
    "error_message": "Tool execution failed: terminal command exited with code 1",
    "stack_trace": "",
    "context": {}
  }' | jq
```

**预期**：返回 `diagnostic`（诊断结果）和 `plan`（含 `strategy`、`actions`、`risk_level` 等）。

## 二、WebSocket 自愈流程测试

WS 用于完整流程：诊断 → 计划 → 执行（流式推送进度）。

### 用 Python 快速验证

```python
# test_self_healing_ws.py
import asyncio
import json
from websockets.sync.client import connect

def test_heal():
    with connect("ws://localhost:8765/ws/self-healing") as ws:
        # 发起一次自愈
        ws.send(json.dumps({
            "type": "heal",
            "error_message": "No tool call found in response",
            "stack_trace": "",
            "context": {},
            "auto_confirm": True   # 无需人工确认即执行
        }))
        # 收流式消息
        while True:
            msg = json.loads(ws.recv())
            print(msg)
            if msg.get("type") in ("completed", "failed", "error"):
                break

if __name__ == "__main__":
    test_heal()
```

运行：`pip install websockets && python test_self_healing_ws.py`

### WS 消息类型说明

- 客户端 → 服务端：
  - `{"type": "heal", "error_message": "...", "stack_trace": "...", "context": {}, "auto_confirm": true}`：发起自愈
  - `{"type": "get_statistics"}`：获取统计
  - `{"type": "get_history", "count": 10}`：获取最近修复历史
- 服务端 → 客户端：`status`、`diagnostic`、`plan`、执行进度、`completed` / `failed` / `error`

## 三、通过 EventBus 触发（端到端）

自愈会由 **工具失败/解析失败** 等事件自动触发，并由 `SelfHealingWorker` 通过 WebSocket 广播到对应 session。

**测试步骤**：

1. 用 Mac 客户端或任意已连接 `/ws` 的会话建立连接（带 `session_id`）。
2. 在对话中**故意制造工具解析失败**，例如：
   - 使用容易导致「No tool call found」的模型或 prompt；
   - 或临时改坏某个工具的 schema，让解析必然失败。
3. 连续触发几次解析失败（注意 Worker 有熔断：默认 3 次后 60 秒内不再触发）。
4. 在客户端或 `/ws` 收包中查看是否收到自愈相关推送（如 `self_healing_status`、诊断/计划等）。

**审核要点**（与文档 12.9 一致）：工具失败后，当前 session 是否收到自愈/升级建议的广播。

## 五、常见问题

- **端口**：确认后端是 `8765`，或修改上述 URL 中的端口。
- **认证**：若后台开启了隧道/API 认证，curl 和 WS 需带相应 header 或 token。
- **熔断**：SelfHealingWorker 默认同一 session 在 60 秒内解析失败 ≥3 次会暂停触发自愈，等 cooldown 后再测或重启后端。
